# Phase 4.5.3: Dashboard Tile Redesign - Research

**Researched:** 2026-03-28
**Domain:** Vanilla JS / CSS — project card UI redesign, phase topology layout, string-based phase parsing
**Confidence:** HIGH

## Summary

This phase replaces the current `.mini-timeline` linear dot strip on dashboard project cards with a two-zone display: a compressed history strip (completed phases as proportional colored rectangles with branching topology) and a stacked frontier list (active/future phases showing full slug names and goal excerpts). The implementation is entirely in `public/index.html` (single-file SPA) using vanilla JS/CSS and the existing theme token system from Phase 4.5.2.

The phase has one non-trivial server-side dependency: the dashboard API currently does not return `requirementCount` per phase (needed for rectangle height scaling) nor ROADMAP.md goal text per phase (needed for frontier goal excerpts). Both need to be added to `buildPhaseList()` in `server.js` and propagated through the `/api/dashboard` response.

The primary implementation challenge is CSS layout for the two-tier branching history strip — depth-1 sub-phases trailing beside their integer parent, depth-2 sub-phases dropping a row below. This is naturally expressed with CSS Grid or nested flex with absolute/relative positioning, but requires visual iteration with real project data.

**Primary recommendation:** Build in two sequential plans — Plan 01 adds `requirementCount` and goal text to the API, then Plan 02 rebuilds the card's mini-timeline zone with the new layout. The A/B iteration step should be a named task in Plan 02 before committing final pixel values.

### Locked Decisions

**History strip — graph-style with branching**
- Completed phases render as colored rectangles (not circles), with no numbers or labels at rest
- Hover tooltip shows phase name, slug, and plan/requirement counts
- Sub-phases cluster beside their parent as smaller rectangles trailing to the right
- Depth tiers stack: depth-1 children (4.5, 4.6) trail on tier 1, depth-2 grandchildren (4.5.1, 4.5.2) drop to tier 2 below their parent sub-phase
- Visual metaphor: `git log --oneline --graph --all --decorate` — branching topology preserved even in compressed form
- Status colors: green (complete), blue (active/in-progress), gray (pending) — use theme tokens from Phase 4.5.2

**Rectangle sizing — proportional by plan count x requirement count**
- Width scales by plan count (px per plan, starting point ~8px/plan)
- Height scales by requirement count (px per requirement, starting point ~6px/req)
- Area = visual proxy for total effort (plans x requirements)
- Minimum size 12x12px ensures 0-plan/0-req phases are still visible
- Sub-phase rectangles render at 75% scale of their parent tier
- Pixel values are starting points — visual iteration with real rendered output is required; A/B testing step planned before finalizing

**Frontier display — stacked list with full names**
- Boundary: first non-complete phase starts the frontier (everything complete → history strip)
- Each frontier phase shows: status dot + full slug name + goal excerpt (~40 chars from ROADMAP.md goal)
- Multiple phases can be in-progress simultaneously — each carries its own status dot
- Sub-phases indent under their parent in the frontier list, mirroring history strip depth tiers
- Pending phases with no plans show name + goal only; in-progress phases show plan progress too

**Phase number parsing**
- Current `isSubPhase()` uses `String(numStr).includes('.')` — only detects 1 level
- Must distinguish depth levels: `4` (depth 0), `4.5` (depth 1), `4.5.1` (depth 2)
- `getParentPhaseNum()` currently splits on first `.` only — needs to return immediate parent (4.5.1 → 4.5, not 4)
- No `parseFloat` for comparison — use string splitting to preserve full precision

### Pattern 1: Phase Depth Classification
**What:** Replace `isSubPhase()` and `getParentPhaseNum()` with depth-aware helpers
**When to use:** Everywhere phase topology is needed (card, detail timeline)

```javascript
// Replace existing isSubPhase() / getParentPhaseNum() in index.html
function getPhaseDepth(numStr) {
  // "4" → 0, "4.5" → 1, "4.5.1" → 2
  const parts = String(numStr).split('.');
  return parts.length - 1;
}

function getImmediateParent(numStr) {
  // "4.5.1" → "4.5", "4.5" → "4", "4" → null
  const parts = String(numStr).split('.');
  if (parts.length <= 1) return null;
  return parts.slice(0, -1).join('.');
}

function phaseKey(numStr) {
  // String identity key — never parseFloat
  return String(numStr);
}
```

### Pattern 2: Depth-Aware Phase Grouping
**What:** Upgrade `groupPhases()` to build a tree rather than flat groups
**When to use:** Both card history strip and frontier list

```javascript
function buildPhaseTree(phases) {
  // Returns array of tree nodes, each with .children[]
  // Input phases sorted by numStr (string sort safe after proper comparison)
  const byKey = {};
  const roots = [];
  for (const phase of phases) {
    byKey[phase.numStr] = { ...phase, children: [] };
  }
  for (const phase of phases) {
    const parentNum = getImmediateParent(phase.numStr);
    if (parentNum && byKey[parentNum]) {
      byKey[parentNum].children.push(byKey[phase.numStr]);
    } else {
      roots.push(byKey[phase.numStr]);
    }
  }
  return roots;
}
```

### Pattern 3: History Strip Rectangle Sizing
**What:** Map plan/requirement counts to pixel dimensions
**When to use:** Rendering each rectangle in the history strip

```javascript
function rectDimensions(planCount, reqCount, scale) {
  // scale: 1.0 for depth-0, 0.75 for depth-1/2 (per CONTEXT.md)
  const minW = 12, minH = 12;
  const pxPerPlan = 8;   // starting point — subject to A/B iteration
  const pxPerReq  = 6;   // starting point — subject to A/B iteration
  const w = Math.max(minW, (planCount || 0) * pxPerPlan) * scale;
  const h = Math.max(minH, (reqCount  || 0) * pxPerReq)  * scale;
  return { width: Math.round(w), height: Math.round(h) };
}
```

### Pattern 4: Frontier Boundary Detection
**What:** Split the allPhases array into history (all-complete prefix) vs. frontier (first non-complete onward)
**When to use:** Rendering card zones

```javascript
function splitHistoryFrontier(phases) {
  // phases: flat sorted array (depth-0 first, then sub-phases interleaved)
  let boundaryIdx = phases.length; // default: all history
  for (let i = 0; i < phases.length; i++) {
    if (phases[i].status !== 'complete') {
      boundaryIdx = i;
      break;
    }
  }
  return {
    history: phases.slice(0, boundaryIdx),
    frontier: phases.slice(boundaryIdx),
  };
}
```

### Pattern 5: String-Safe Phase Sorting
**What:** Sort phases by numStr without parseFloat (which collapses 4.5.1 and 4.5.10)
**When to use:** Everywhere phases are sorted — replaces ALL `parseFloat()` comparisons

```javascript
function comparePhaseNums(a, b) {
  // Compare "4.5.1" vs "4.5.2" segment by segment as integers
  const partsA = String(a).split('.').map(Number);
  const partsB = String(b).split('.').map(Number);
  const len = Math.max(partsA.length, partsB.length);
  for (let i = 0; i < len; i++) {
    const diff = (partsA[i] || 0) - (partsB[i] || 0);
    if (diff !== 0) return diff;
  }
  return 0;
}
// Usage: phases.sort((a, b) => comparePhaseNums(a.numStr, b.numStr));
```

### Anti-Patterns to Avoid

- **`parseFloat('4.5.1')` for comparison:** Returns `4.5` — loses the third segment entirely. Any sort or dedup using `parseFloat` on multi-segment numStr is wrong.
- **Rebuilding the detail timeline:** `renderTimeline()` and `makePhaseNode()` are for the project detail page, NOT the card. Phase 4.5.3 only touches the card's mini-timeline zone.
- **Using absolute px in CSS for rectangle sizes:** Rectangle dimensions are set inline via JS (`element.style.width`), not CSS classes — CSS only provides the structural skeleton and status colors.
- **Single-depth `getParentPhaseNum()`:** The existing function does `split('.')[0]` which maps 4.5.1 → 4, not 4.5. Must be replaced.

## Common Pitfalls

### Pitfall 1: parseFloat Truncation on Multi-Segment Phase Numbers
**What goes wrong:** `parseFloat('4.5.1')` returns `4.5`, making 4.5.1 appear identical to 4.5 in dedup Sets and sort comparisons. Phases get dropped or mis-sorted.
**Why it happens:** The codebase uses `parseFloat(numStr)` in three places in `index.html` (lines 1325, 1326, 1328, 1329, 1331, 1859, 1861, 1864) and once in `server.js` `parsePhaseDir()` (`num: parseFloat(m[1])`). The dashboard card and detail page both have this bug.
**How to avoid:** Replace all `parseFloat` comparisons with `comparePhaseNums()`. The `num` field in `parsePhaseDir()` (server.js) is used only for sorting — it can stay as a float since current phase numbers don't exceed two decimal segments, but the client-side code must be fixed.
**Warning signs:** Two phases with numStr "4.5" and "4.5.1" where dedup silently drops one.

### Pitfall 2: Card Height Explosion
**What goes wrong:** Adding a branching history strip (two rows) plus a frontier list below the existing callout zone makes cards much taller, breaking the horizontal-aspect grid layout.
**Why it happens:** The card grid uses `minmax(320px, 1fr)` with no explicit height constraint; content drives height.
**How to avoid:** The history strip should use `overflow: hidden` with a max-height, or the frontier list should be capped (e.g., 3-4 visible items with scroll). Discretion area — but plan for it explicitly in the A/B iteration step.
**Warning signs:** Dashboard cards grow to 3x the height of cards with no frontier phases.

### Pitfall 3: Branching CSS Layout with Variable-Width Rectangles
**What goes wrong:** Sub-phase rectangles need to align horizontally with their parent rectangle's position, but parent widths are variable (proportional to plan count). Simple `margin-left: auto` or fixed offsets won't align sub-phases under their correct parent.
**Why it happens:** The history strip is not a fixed grid — rectangle widths vary. The sub-phase row must flow under the correct parent position.
**How to avoid:** Use a container-per-parent-group pattern. Each parent rectangle and its trailing sub-phases are wrapped in a `.history-parent-group` flex container. The parent rect is first; sub-phase tiers are stacked below it inside that group. No global alignment math needed.
**Warning signs:** Sub-phase rectangles drift visually away from their parent when parent widths change.

### Pitfall 4: API Missing requirementCount and Goal Text
**What goes wrong:** Rectangle height (by requirement count) and frontier goal excerpt rely on data not currently in `/api/dashboard` response. Frontend renders minimum-height rectangles for everything and frontier items show no goals.
**Why it happens:** `buildPhaseList()` returns `planCount` and `completedPlans` but not `requirementCount`. `parseRoadmapPhaseNames()` returns names but not goals. The dashboard endpoint does not include ROADMAP.md goal text.
**How to avoid:** Plan 01 must add both fields before Plan 02 does the frontend work.
**Warning signs:** All rectangles render at minimum 12×12 even for phases with 5+ requirements.

### Pitfall 5: parseRoadmapPhaseNames Regex Skips Depth-2 Phases
**What goes wrong:** The regex `Phase\s+(\d+(?:\.\d+)?)\s*:` only matches one optional decimal group — it captures "4.5" but NOT "4.5.1".
**Why it happens:** `\d+(?:\.\d+)?` is "one decimal group, optional" — a single repetition, not one-or-more.
**How to avoid:** Change to `\d+(?:\.\d+)*` in `parseRoadmapPhaseNames()`. Same fix needed on the client-side in any analogous regex.
**Warning signs:** Frontier items for depth-2 phases (4.5.1, 4.5.3) show empty goal excerpts even when ROADMAP.md has them.

Two fields missing from the API that this phase needs:

### 1. requirementCount per phase
**Where to add:** `buildPhaseList()` and `buildPhaseListFromReader()` in `server.js` — scan each phase directory for plan files, parse their `requirements:` frontmatter with `parsePlanFrontmatter()`, and deduplicate requirement IDs.

```javascript
// In getPhaseInfo() or buildPhaseList() — after planDetails is built
const allReqs = new Set(info.planDetails.flatMap(p => p.requirements || []));
return { ..., requirementCount: allReqs.size };
```

### 2. Goal text per phase (for frontier display)
**Where to add:** `parseRoadmapPhaseNames()` needs a companion `parseRoadmapPhaseGoals()` function that extracts the `**Goal:**` line following each `### Phase X:` heading.

```javascript
function parseRoadmapPhaseGoals(content) {
  const goals = {};
  const lines = content.split('\n');
  let currentPhase = null;
  for (const line of lines) {
    const mHeading = line.match(/^###\s+Phase\s+(\d+(?:\.\d+)*)\s*:/);
    if (mHeading) { currentPhase = mHeading[1]; continue; }
    if (currentPhase) {
      const mGoal = line.match(/^\*\*Goal[:\*]*\s*(.+)/);
      if (mGoal) {
        goals[currentPhase] = mGoal[1].trim();
        currentPhase = null;
      }
    }
  }
  return goals;
}
```

Then propagate `phaseGoals` alongside `phaseNames` in the dashboard and detail API responses.
