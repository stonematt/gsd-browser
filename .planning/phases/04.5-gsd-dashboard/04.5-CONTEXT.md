# Phase 4.5: GSD Dashboard - Context

**Gathered:** 2026-03-22
**Status:** Ready for planning

<domain>
## Phase Boundary

A multi-project dashboard that surfaces GSD progress, editorial context, and phase documentation across registered sources and branches. The dashboard replaces the landing page and serves as the primary entry point to the app. It parses `.planning/STATE.md` and `ROADMAP.md` from registered sources to show project status, phase timelines, and rendered phase documentation. Non-GSD sources appear in a separate section with fallback to file browser. Branch awareness reads `.planning/` from non-default branches via `git show`.

</domain>

<decisions>
## Implementation Decisions

### Dashboard placement and navigation
- Dashboard replaces the homepage at `/` — it is the front door to the app
- Persistent header bar with navigation links: Dashboard (home), Browse (file browser), Sources (management page)
- Two-level navigation: clicking a GSD project card opens a project detail page (with phase timeline), not directly to the file browser
- Non-GSD sources (no `.planning/STATE.md`) appear in a separate "Other Sources" section below GSD project cards, linking directly to the file browser

### Project card design
- Phase dots/chips for progress visualization — small colored indicators per phase (green=done, blue=active, gray=pending) showing individual phase status at a glance
- Card content: project name, phase dots, current focus line from STATE.md, last activity date from STATE.md
- Responsive CSS grid layout (2-3 columns on wide screens, single column on narrow)
- Quick-links to PROJECT.md, STATE.md, ROADMAP.md on each card — one click to render

### Phase timeline and detail page
- Horizontal bar with circle/dot nodes for each phase, connected by a line, color-coded by status
- Each phase node shows: phase name below, plan count progress (e.g., "2/3 plans")
- Split view layout on detail page: timeline fixed at top, documentation panel below
- Documentation panel uses file list + rendered content pane (two-column within the panel) — left side lists phase files (PLAN.md, RESEARCH.md, SUMMARY.md, etc.), right side renders selected file
- Clicking a phase in the timeline updates the file list to show that phase's directory contents

### Branch awareness
- Dashboard card shows default branch progress only, with a branch count badge (e.g., "3 branches") when other branches have `.planning/`
- Branch discovery is on-demand — only when user opens the project detail page, not at dashboard load
- Branch dropdown above the timeline on the detail page — switching branches reloads timeline and docs for that branch's `.planning/` state
- Branch data read via `git show <branch>:.planning/STATE.md` and similar git commands
- Graceful silent fallback — if git operations fail (not a repo, git not installed, permissions), branch UI simply doesn't appear; no error messages

### Claude's Discretion
- Quick-link visual style (inline text links vs icon buttons vs badges)
- Phase node size, spacing, and active-phase indicator style (pulse, ring, etc.)
- Documentation panel proportions and file list styling
- How to parse STATE.md frontmatter (YAML parsing approach)
- API endpoint design for dashboard data aggregation
- Single-branch repos: whether to show branch badge at all or only when >1 branch
- Loading states and transitions between dashboard → detail → docs

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `public/index.html`: Current browse SPA with dark theme, three-panel layout, hash routing, source switcher. Dashboard page will replace this but can reuse the color palette and CSS patterns.
- `public/sources.html`: Source management page with modals and table layout. Stays as-is; linked from persistent header nav.
- `public/styles/markdown.css`: Existing markdown rendering stylesheet. Reused in the documentation content pane on the detail page.
- `/render?path=...&fragment=true`: Fragment rendering mode already exists — returns just the `<div class="markdown-body">` content. Perfect for injecting rendered docs into the detail page panel.
- `/api/sources`: Returns all registered sources with enrichment. Dashboard can use this to get the source list.
- `/api/sources/:name/tree`: Returns recursive file tree for a source. Can be used to list phase directory contents.
- `buildTree()` in `server.js`: Recursive tree builder that filters to .md files only. Could be adapted or reused for phase directory listing.

### Established Patterns
- CJS modules throughout (`require`/`module.exports`)
- Fastify route pattern with per-route CSP via `onSend` hook
- `MANAGEMENT_CSP` allows inline scripts — dashboard pages need this
- `activeSources` mutable reference in server closure, updated by add/remove
- Single-file SPA with inline style/script — no build tooling (index.html, sources.html pattern)
- Dark theme colors: `#0d1117` (bg), `#161b22` (surface), `#21262d` (border), `#c9d1d9` (text), `#f0f6fc` (bright text), `#58a6ff` (link), `#8b949e` (muted)

### Integration Points
- `public/index.html` — Replace with dashboard page (same file path, new content); existing browse UI content moves to a new route or becomes a view within the SPA
- `src/server.js` — New API endpoints needed: `/api/dashboard` (aggregated project data), `/api/projects/:name/detail` (timeline + phase files), `/api/projects/:name/branches` (branch list with .planning/)
- `src/sources.js` `listSources()` — Source list is the starting point for dashboard data; needs enrichment with STATE.md parsing
- Git operations — New server-side logic to run `git` commands for branch discovery and cross-branch file reads

</code_context>

<specifics>
## Specific Ideas

- Dashboard is the natural front door since this tool exists specifically to browse GSD artifacts
- Phase dots on cards give an at-a-glance project health view without needing to click in
- Split view on detail page (timeline top, docs bottom) keeps the phase context visible while reading documentation
- File list + content pane in the docs panel mirrors the existing file browser pattern — consistency for the user
- Branch awareness is additive — everything works without git, branches are a bonus when available

</specifics>
