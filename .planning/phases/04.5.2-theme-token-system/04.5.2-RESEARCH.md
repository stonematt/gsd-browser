## Summary

This phase is a focused color-token extraction and Catppuccin palette migration. The codebase currently uses GitHub-dark hex values scattered across `index.html` (inline `<style>`), `sources.html` (inline `<style>`), `src/renderer.js` (2 inline style attributes in a template string), and `public/styles/markdown.css` (numerous hardcoded hex values plus a few upstream `var(--*)` references that currently have no values defined). The goal is to create a single `public/styles/theme.css` that defines all color tokens and then replace every hardcoded hex value in all four files with `var(--token-name)`.

The Catppuccin palette is well-documented with exact official hex values. The mapping from current GitHub-dark colors to Catppuccin Mocha is mostly straightforward — the hex values are similar but not identical, so a deliberate visual shift is expected and acceptable. The light theme (Catppuccin Latte) is a full palette designed for readability on light backgrounds and follows the same semantic color roles as Mocha.

The technical approach is pure CSS — no JavaScript, no build tools, no external dependencies. CSS `@media (prefers-color-scheme: light)` overriding `:root` variables is the standard browser-native approach and is supported by all modern browsers. The existing CSP allows `'unsafe-inline'` styles on management routes, so inline `var()` references in HTML will work without any CSP changes.

**Primary recommendation:** Create `theme.css` with `:root` (Mocha) and `@media (prefers-color-scheme: light)` (Latte) blocks, then do a systematic search-and-replace of all hardcoded hex values across the four affected files.

### Pattern 1: Token File Structure (`:root` + media query override)

**What:** Define all tokens in `:root` for the dark default. Override only the values that differ in the light theme inside `@media (prefers-color-scheme: light)`.
**When to use:** Always. This is the canonical browser-native theming pattern.

```css
/* Source: MDN Web Docs — CSS Custom Properties + prefers-color-scheme */

/* === DARK THEME (default: Catppuccin Mocha) === */
:root {
  /* Backgrounds */
  --bg-page:    #1e1e2e;  /* Mocha Base */
  --bg-surface: #181825;  /* Mocha Mantle */
  --bg-overlay: #313244;  /* Mocha Surface 0 */

  /* Borders */
  --border-default: #45475a;  /* Mocha Surface 1 */
  --border-muted:   #313244;  /* Mocha Surface 0 */

  /* Text */
  --text-primary:   #cdd6f4;  /* Mocha Text */
  --text-secondary: #bac2de;  /* Mocha Subtext 1 */
  --text-muted:     #6c7086;  /* Mocha Overlay 0 */
  --text-link:      #89b4fa;  /* Mocha Blue */
  --text-bright:    #cdd6f4;  /* Mocha Text (same as primary here) */

  /* Status */
  --status-complete:      #a6e3a1;  /* Mocha Green */
  --status-active:        #89b4fa;  /* Mocha Blue */
  --status-pending:       #6c7086;  /* Mocha Overlay 0 */
  --status-complete-text: #a6e3a1;
  --status-active-text:   #89b4fa;
```

### Pattern 2: Consumer File Linking

**What:** Every HTML page that uses color tokens links `theme.css` in `<head>` before any inline `<style>` blocks.
**When to use:** All HTML consumers. The file must load before inline styles that reference its variables.

```html
<head>
  <link rel="stylesheet" href="/styles/theme.css">
  <link rel="stylesheet" href="/styles/markdown.css">
  <style>
    /* Layout and structure only — no hardcoded hex values */
    body { background: var(--bg-page); color: var(--text-primary); }
  </style>
</head>
```

### Pattern 3: renderer.js Page Template

**What:** The breadcrumb template in `renderer.js` currently injects inline styles with hardcoded hex. Replace with `var()` references — inline styles in rendered HTML support CSS custom properties if the token is defined in a linked stylesheet.
**When to use:** The rendered page template in `buildPage()` links `theme.css`, so `var(--token-name)` works in inline styles.

```javascript
// Current (in renderer.js template string):
`<a href="/" style="color:#58a6ff;text-decoration:none">&larr; Home</a>`
`<span style="color:#8b949e;margin:0 6px">/</span>`

// Replace with:
`<a href="/" style="color:var(--text-link);text-decoration:none">&larr; Home</a>`
`<span style="color:var(--text-muted);margin:0 6px">/</span>`
```

**Alternative (discretion):** Add a `.breadcrumb-home-link` and `.breadcrumb-sep` class to `markdown.css` gsd-browser overrides section, then use class names in the template instead of inline styles. This is cleaner but requires an extra step.

### Pattern 4: markdown.css Existing `var()` References

The file already references `var(--fgColor-danger)`, `var(--bgColor-danger-muted)`, `var(--borderColor-accent-emphasis)`, `var(--borderColor-muted)`. These are GitHub-style variable names with no values currently provided — they resolve to `initial` (invalid, so the property is effectively unset). Two options:

1. **Provide values for those specific names** in `theme.css` using GitHub's naming convention (e.g., `--borderColor-muted: var(--border-muted)`). This is a bridge pattern.
2. **Replace those references in markdown.css** with the new semantic token names.

Option 2 is cleaner and eliminates the GitHub-style naming entirely. It is recommended.

### Anti-Patterns to Avoid

- **Defining tokens per-component instead of centrally:** Every token must live only in `theme.css`. Inline overrides defeat the whole system.
- **Using `color-scheme` property as the theming mechanism:** `color-scheme` affects browser UA styles (scrollbars, form inputs). It is not a substitute for `prefers-color-scheme` media queries. The existing `color-scheme: dark` on `.markdown-body` in `markdown.css` should stay — it tells the browser to style scrollbars and form inputs for dark mode.
- **Duplicating all tokens in the light override:** Only override the values that change. Since all color values change for Latte, the full set must be in the override block, but this is expected.
- **Forgetting rgba() occurrences:** Several colors appear as `rgba(88, 166, 255, 0.15)` (hover glow) rather than hex. These must also be replaced. Use `color-mix(in srgb, var(--status-active) 15%, transparent)` or define dedicated semi-transparent tokens like `--status-active-glow: rgba(137, 180, 250, 0.15)`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Color interpolation for rgba variants | Custom JS to compute rgba from hex | `color-mix()` or define named glow tokens | Browser-native, readable, maintainable |
| Theme detection | JS that reads `window.matchMedia` and sets classes | `@media (prefers-color-scheme: light)` in CSS | No JS needed; OS change triggers automatically |
| Cross-file token sharing | Copying token values into each file | Single `theme.css` linked by all consumers | Single source of truth |

**Key insight:** Every part of this phase is pure CSS authoring. There is no library, framework, or tool to install.

## Common Pitfalls

### Pitfall 1: rgba() values not tokenized

**What goes wrong:** After replacing hex values, several rgba() values remain hardcoded (e.g., `rgba(88, 166, 255, 0.5)` for the active phase glow ring, `rgba(31, 111, 235, 0.12)` for active tree-file background). These won't respond to theme changes.
**Why it happens:** rgba() values are visually distinct from hex and easy to miss in a search.
**How to avoid:** Search for `rgba(` in all affected files and handle each occurrence. Define semi-transparent tokens for the repeated ones.
**Warning signs:** Active/selected states look wrong in light theme.

### Pitfall 2: renderer.js page template doesn't link theme.css

**What goes wrong:** The rendered markdown page shows hardcoded colors because `theme.css` is not in the `<head>` of `buildPage()`.
**Why it happens:** The template string in `renderer.js` is separate from `index.html` and `sources.html`.
**How to avoid:** Verify `buildPage()` includes `<link rel="stylesheet" href="/styles/theme.css">` in the `<head>`. The `markdown.css` link is already there (visible in `markdown.css` gsd-browser overrides); add `theme.css` before it.
**Warning signs:** Breadcrumb links in rendered markdown pages show wrong colors.

### Pitfall 3: markdown.css `color-scheme: dark` conflicts with light theme

**What goes wrong:** `color-scheme: dark` on `.markdown-body` tells the browser to render scrollbars and form inputs in dark style even when Latte is active.
**Why it happens:** `color-scheme` is independent of `prefers-color-scheme` — it's a declaration about which color schemes the element supports.
**How to avoid:** Change `.markdown-body { color-scheme: dark; }` to `.markdown-body { color-scheme: dark light; }` to declare support for both.
**Warning signs:** Form checkboxes in task lists appear inverted in light mode.

### Pitfall 4: Incomplete replacement in markdown.css syntax-highlight token colors

**What goes wrong:** The `.pl-*` syntax highlight classes in `markdown.css` (`.pl-c`, `.pl-c1`, `.pl-e`, etc.) use hardcoded hex for code highlight colors. These are GitHub's Primer syntax colors — they are code-specific and may not map to Catppuccin semantic tokens.
**Why it happens:** The `.pl-*` classes serve a different purpose (syntax highlighting) than UI chrome colors. They were not listed in the CONTEXT.md inventory.
**How to avoid:** These classes affect code syntax colors in non-Shiki fallback rendering. Since Shiki renders its own inline styles (and handles its own theming), and the Shiki theme is hardcoded to `github-dark` in `renderer.js`, the `.pl-*` classes are only active in the fallback path. Treat them as out of scope for this phase — they're not UI chrome. Leave them with their hardcoded GitHub colors. Document this decision.
**Warning signs:** N/A — the `.pl-*` colors only appear when Shiki fails to load, which is a degraded state.

### Pitfall 5: Missing `<link>` in sources.html

**What goes wrong:** `sources.html` currently has no external CSS links — all its styles are inline. Adding `theme.css` requires adding a `<link>` tag in its `<head>`.
**Why it happens:** The file was built self-contained.
**How to avoid:** Explicitly add `<link rel="stylesheet" href="/styles/theme.css">` to `sources.html`'s `<head>` as the first task.
**Warning signs:** sources.html renders with wrong colors or undefined variables.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `prefers-color-scheme` + JS class toggle | `@media (prefers-color-scheme: light)` in CSS only | CSS spec evolution | No JS needed for auto switching |
| Per-component hardcoded colors | `:root` custom property tokens | Modern CSS practice | Single-file theme changes |
| `color()` function (P3 gamut) | hex values | Catppuccin uses hex | Simpler, universally supported |

**Note on `color-mix()`:** `color-mix(in srgb, var(--text-link) 15%, transparent)` is an alternative to hardcoded rgba tokens for the semi-transparent variants. Browser support is Firefox 113+, Chrome 111+, Safari 16.2+. This is usable for a developer tool targeting modern browsers. Either approach works; named rgba tokens are simpler and more readable.

1. **`.pl-*` syntax highlight colors in markdown.css**
   - What we know: These 20+ color values are GitHub Primer syntax token colors, only active in the Shiki fallback path
   - What's unclear: Are they ever visible in normal operation?

   - What we know: Currently set to `color-scheme: dark` which affects browser UA rendering (scrollbars, checkboxes)
