# Phase 4.5.3: Dashboard Tile Redesign - Context

**Gathered:** 2026-03-28
**Status:** Ready for planning

<domain>
## Phase Boundary

Project cards on the dashboard communicate project shape at a glance. Completed phases compress into a graph-style history strip with proportional rectangles. Active and future phases expand into a stacked frontier list with full slug names and goal excerpts. Sub-phases branch visually at their correct depth tier. Phase number parsing correctly distinguishes depth levels (4 vs 4.5 vs 4.5.1).

</domain>

<decisions>
## Implementation Decisions

### History strip — graph-style with branching
- Completed phases render as colored rectangles (not circles), with no numbers or labels at rest
- Hover tooltip shows phase name, slug, and plan/requirement counts
- Sub-phases cluster beside their parent as smaller rectangles trailing to the right
- Depth tiers stack: depth-1 children (4.5, 4.6) trail on tier 1, depth-2 grandchildren (4.5.1, 4.5.2) drop to tier 2 below their parent sub-phase
- Visual metaphor: `git log --oneline --graph --all --decorate` — branching topology preserved even in compressed form
- Status colors: green (complete), blue (active/in-progress), gray (pending) — use theme tokens from Phase 4.5.2

### Rectangle sizing — proportional by plan count × requirement count
- Width scales by plan count (px per plan, starting point ~8px/plan)
- Height scales by requirement count (px per requirement, starting point ~6px/req)
- Area = visual proxy for total effort (plans × requirements)
- Minimum size 12×12px ensures 0-plan/0-req phases are still visible
- Sub-phase rectangles render at 75% scale of their parent tier
- **Pixel values are starting points** — visual iteration with real rendered output is required; A/B testing step planned before finalizing

### Frontier display — stacked list with full names
- Boundary: first non-complete phase starts the frontier (everything complete → history strip)
- Each frontier phase shows: status dot + full slug name + goal excerpt (~40 chars from ROADMAP.md goal)
- Multiple phases can be in-progress simultaneously — each carries its own status dot
- Sub-phases indent under their parent in the frontier list, mirroring history strip depth tiers
- Pending phases with no plans show name + goal only; in-progress phases show plan progress too

### Phase number parsing
- Current `isSubPhase()` uses `String(numStr).includes('.')` — only detects 1 level
- Must distinguish depth levels: `4` (depth 0), `4.5` (depth 1), `4.5.1` (depth 2)
- `getParentPhaseNum()` currently splits on first `.` only — needs to return immediate parent (4.5.1 → 4.5, not 4)
- No `parseFloat` for comparison — use string splitting to preserve full precision

### Visual iteration process
- Implementation should include an A/B testing step with real project data
- Pixel scaling values (8px/plan, 6px/req) are tunable starting points
- Expect multiple rounds of visual inspection to balance rectangle proportions, spacing, and card layout

### Claude's Discretion
- Connector line style between history strip rectangles (solid, dashed, or gap-only)
- Exact border-radius on rectangles (likely 3-4px)
- Hover tooltip styling and content formatting
- Frontier list typography (font sizes, line height, goal excerpt truncation)
- How to handle phases with 0 requirements mapped (use plan count only, or minimum height)
- Card layout adjustments to fit history strip + frontier list (may need vertical stacking within card)
- Spacing between history rectangles and between frontier list items

</decisions>

## Specific Ideas

- "Real estate efficiency is key on the dashboard — cards are horizontal aspect ratio"
- "Words over numbers — the feature/deliverable/objective gives substance, not the ordinal"
- "It's almost like git log --oneline --graph --all --decorate"
- "Requirements are related to human/user effort and proxy for value to the product"
- "Plan count already implicitly encodes difficulty — harder phases get more plans"
- Width × height rectangle sizing: "let's try this idea, I have high hopes"
- Visual iteration expected: "I'm going to need to see some real non-ASCII requirements"
