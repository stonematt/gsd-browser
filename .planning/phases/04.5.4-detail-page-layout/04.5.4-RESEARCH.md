## Summary

Phase 4.5.4 replaces the current horizontal phase timeline (`#phase-timeline-container`) and 2-column docs panel (`#docs-panel`) with a three-column CSS grid layout. The work is almost entirely client-side DOM/CSS in `public/index.html`, plus one small server-side change to strip YAML frontmatter in `/render`.

The existing codebase provides nearly everything needed: `buildPhaseTree()`, `getPhaseDepth()`, `comparePhaseNums()`, `splitHistoryFrontier()`, `classifyPhaseFile()`, `loadPhaseFile()`, and the plan metadata card are all available in `index.html`. The server already parses frontmatter separately via `parsePlanFrontmatter()`, so stripping it from the rendered output is a one-liner. The Catppuccin theme token system from phase 4.5.2 covers all color needs without hardcoding hex values.

The design reference at `.planning/research/detail-page-design-reference.md` is the authoritative art direction spec and provides exact dimensions, color tokens, and interaction flows. All implementation decisions are already locked in CONTEXT.md.

**Primary recommendation:** Implement as two plans — Plan 01: server-side frontmatter stripping + three-column grid HTML/CSS structure; Plan 02: vertical navigator JS logic (completed collapse, dot scaling, connectors, localStorage persistence).

### Pattern 6: File List Grouping (Col 2)

The file list must be regrouped. Currently it just lists all `.md` files. New behavior:

```javascript
function groupPhaseFiles(files) {
  // paired above-divider: PLAN then SUMMARY for each plan number
  const planNums = [...new Set(
    files.filter(f => f.endsWith('-PLAN.md'))
         .map(f => f.match(/^(\d+-\d+)/)?.[1])
         .filter(Boolean)
  )].sort();
  const above = [];
  for (const num of planNums) {
    const plan = files.find(f => f === num + '-PLAN.md');
    const summary = files.find(f => f === num + '-SUMMARY.md');
    if (plan) above.push(plan);
    if (summary) above.push(summary);
  }
  // below-divider: CONTEXT, RESEARCH, VALIDATION (everything else)
  const above_set = new Set(above);
  const below = files.filter(f => f.endsWith('.md') && !above_set.has(f));
  return { above, below };
}
```

### Anti-Patterns to Avoid

- **Hardcoding hex colors:** Always use CSS vars from `theme.css`. Never write `#3fb950` directly in index.html.
- **Using numStr for display:** Use `phase.displayNum || phase.numStr` for text — `normalizePhaseNum()` ensures "4.5.4" not "04.5.4".
- **Letting col 1 content bleed during collapse:** `#phase-navigator` needs `overflow: hidden` AND a fixed `width: 200px` to prevent content from reflowing as the grid column shrinks.
- **Building a new `groupPhases()` for the detail page:** The existing `groupPhases()` is a legacy adapter for `renderTimeline()`. The new navigator renders from `buildPhaseTree()` directly — don't reuse the adapter.
- **Re-inventing the metadata card:** `loadPhaseFile()` already injects `.plan-meta-card` for PLAN.md files. The card construction code stays; only the frontmatter strip is new.

## Common Pitfalls

### Pitfall 1: CSS Grid Transition Requires `overflow: hidden` on the Collapsing Column

**What goes wrong:** When `grid-template-columns` transitions from `200px` to `0px`, the column content doesn't clip automatically — it bleeds over adjacent columns during the animation.
**Why it happens:** Grid tracks collapse but child elements have their own widths.
**How to avoid:** Set `overflow: hidden` on `#phase-navigator` AND give it a fixed `width: 200px` (matching the grid track) so its content respects the shrinking boundary.
**Warning signs:** Column 2 content appears overlaid by column 1 text during the toggle animation.

### Pitfall 2: `currentProjectPhase` State Must Be Updated for the New selectPhase

**What goes wrong:** `selectPhase()` currently stores `currentProjectPhase` and calls `container.dataset.sourcePath` on `#phase-timeline`. After this phase, `#phase-timeline` is gone.
**Why it happens:** The function is tightly coupled to the old DOM IDs.
**How to avoid:** When rewriting `selectPhase()`, store `sourcePath` on `#detail-body` (or a new state variable) — not on the removed `#phase-timeline` container.
**Warning signs:** File list stops populating when a phase is clicked.

### Pitfall 3: `loadProjectDetail()` References Removed DOM IDs

**What goes wrong:** `loadProjectDetail()` references `#phase-timeline` (line 1953), `#phase-file-list` (line 1954), and `#phase-content` (line 1955) for loading states. After replacing the HTML structure, these references break silently.
**Why it happens:** The loading state code was written against the old structure.
**How to avoid:** Update the loading state logic alongside the HTML structure change.
**Warning signs:** Blank screen or JS errors on project navigation.

### Pitfall 4: Frontmatter Regex Must Handle Windows Line Endings

**What goes wrong:** The regex `/^---\n[\s\S]*?\n---\n?/` fails if the file has `\r\n` line endings.
**Why it happens:** PLAN.md files are written by agents on any OS.
**How to avoid:** Either normalize line endings before stripping (`content.replace(/\r\n/g, '\n')`) or make the regex tolerant (`/^---\r?\n[\s\S]*?\r?\n---\r?\n?/`). The tolerant regex is lower risk.
**Warning signs:** Frontmatter appears in the rendered output for files from Windows-origin agents.

### Pitfall 5: Mini Dot Strip in Collapsed Completed Row

**What goes wrong:** The mini dot strip (showing branching shape of completed phases) is the most ambiguous element to implement. Attempting a canvas or SVG approach adds complexity.
**Why it happens:** The spec says "small dots for trunk phases, smaller dots for sub-phases, all in complete color." This is discretionary territory.
**How to avoid:** Implement as a flex row of small `<span>` dots generated from the completed phases list. No SVG needed. Keep dot count capped at ~10 to avoid overflow.
**Warning signs:** Overly complex implementation that blocks the rest of the phase.

### Pitfall 6: Phase Numbers in Navigator Must Use displayNum

**What goes wrong:** Displaying `phase.numStr` ("04.5.4") instead of `phase.displayNum` ("4.5.4") in the navigator text.
**Why it happens:** numStr is the identity key; displayNum is the display form.
**How to avoid:** Always use `phase.displayNum || normalizePhaseNum(phase.numStr)` for any visible text.
**Warning signs:** Phase labels read "04.5.4" with leading zero.

### Existing `buildPhaseTree()` (client-side, index.html ~line 2054)
```javascript
function buildPhaseTree(phases) {
  const byKey = {};
  const roots = [];
  for (const phase of phases) {
    byKey[String(phase.numStr)] = { ...phase, children: [] };
  }
  for (const phase of phases) {
    const parentNum = getImmediateParent(phase.numStr);
    if (parentNum && byKey[parentNum]) {
      byKey[parentNum].children.push(byKey[String(phase.numStr)]);
    } else {
      roots.push(byKey[String(phase.numStr)]);
    }
  }
  return roots;
}
```

### Existing `splitHistoryFrontier()` (client-side, index.html ~line 2071)
```javascript
function splitHistoryFrontier(phases) {
  let boundaryIdx = phases.length;
  for (let i = 0; i < phases.length; i++) {
    if (phases[i].status !== 'complete') {
      boundaryIdx = i;
      break;
    }
  }
  return { history: phases.slice(0, boundaryIdx), frontier: phases.slice(boundaryIdx) };
}
```
