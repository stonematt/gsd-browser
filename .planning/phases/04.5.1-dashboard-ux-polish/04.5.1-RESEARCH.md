### Pattern 1: Timeline Overflow â€” Collapse Completed Phases (DASH-07)

**When to use:** Apply when phase count exceeds a threshold (detect via `phases.filter(p => p.status === 'complete').length > N`) OR use CSS-only responsive approach.

**Mini-timeline on cards:** The card mini-timeline already uses `overflow-x: auto` â€” leave as-is (cards are fixed-width so overflow is bounded). The pending phase indicators (DASH-09) will be appended to this timeline.

### Pattern 2: Sub-Phase Visual Hierarchy (DASH-08)

**What:** Decimal phases (4.5, 4.5.1) render visually subordinate to their parent integer phase, not as equal peers in the timeline row.

**Detection logic (JS):**
```javascript
function isSubPhase(numStr) {
  return numStr.includes('.');
}

function getParentPhaseNum(numStr) {
  return numStr.split('.')[0];
}
```

**Visual approach:** Indent sub-phases below and slightly right of their parent, using a vertical connector. The timeline changes from a pure horizontal row to a row with optional vertical sub-branches.

```css
.phase-node--subphase {
  margin-top: 4px;
  margin-left: 12px;      /* indented under parent */
  transform: scale(0.9);  /* slightly smaller */
  opacity: 0.9;
}

.phase-node--subphase .phase-dot {
  width: 12px;
  height: 12px;  /* smaller than parent 16px */
}

/* Vertical connector from parent to sub-phase group */
.phase-subphase-group {
  display: flex;
  flex-direction: column;
  align-items: flex-start;
}
```

**DOM structure change:** In `renderTimeline()`, group integer phases with their decimal children:
```javascript
// Group phases: { parent: phaseObj, children: [subphaseObj, ...] }
function groupPhases(phases) {
  const groups = [];
  for (const phase of phases) {
    if (!isSubPhase(phase.numStr)) {
      groups.push({ parent: phase, children: [] });
    } else {
      const parentNum = getParentPhaseNum(phase.numStr);
      const group = groups.find(g => g.parent.numStr === parentNum
        || g.parent.numStr.startsWith(parentNum + '.'));
      if (group) {
        group.children.push(phase);
      } else {
        // Orphaned sub-phase â€” render as peer (fallback)
        groups.push({ parent: phase, children: [] });
      }
    }
  }
  return groups;
}
```

**Mini-timeline on cards:** Same grouping â€” sub-phase dots rendered below their parent dot with a smaller size and indent.

### Pattern 3: Pending Phase Indicators on Cards (DASH-09)

**Problem:** The card mini-timeline only shows phases that exist as directories in `.planning/phases/`. Phases defined in ROADMAP.md but not yet started have no directory and are invisible.

**Data gap:** The server's `/api/dashboard` response includes `phaseStatus` from `buildPhaseList()` (filesystem scan), and `dependencies` from `parseRoadmapDeps()`. But ROADMAP.md phase total count isn't surfaced.

**Server fix needed:** In `server.js`, the `/api/dashboard` endpoint should also parse ROADMAP.md phase headers to get the total expected phase list. The `parseRoadmapDeps()` function already parses phase headers â€” it already has `deps` keys for all phases in ROADMAP.md. The deps object keys ARE the phase numStr values.

**Approach:** Use `Object.keys(dependencies)` as the "all expected phases" list. Phases present in `dependencies` but absent from `phaseStatus` are pending/unstarted.

In the frontend `renderProjectCard()`:
```javascript
// Build full phase list: existing phases + unstarted phases from roadmap
const roadmapPhaseNums = Object.keys(project.dependencies || {}).sort((a, b) => parseFloat(a) - parseFloat(b));
const existingNums = new Set(phases.map(p => p.numStr));
const unstartedPhases = roadmapPhaseNums
  .filter(num => !existingNums.has(num))
  .map(num => ({ numStr: num, status: 'pending', planCount: 0, completedPlans: 0 }));
const allPhases = [...phases, ...unstartedPhases].sort((a, b) => parseFloat(a.numStr) - parseFloat(b.numStr));
```

**Confidence:** HIGH â€” `dependencies` object is already in the dashboard API response. No server changes needed.

### Pattern 6: Requirement Chips/Badges (DASH-12)

**Data gap:** `parsePlanFrontmatter()` currently only extracts `wave` and `dependsOn`. The `requirements` YAML array is present in plan files but not parsed.

**Server fix required:** In `server.js`, update `parsePlanFrontmatter`:
```javascript
function parsePlanFrontmatter(content) {
  const result = { wave: null, dependsOn: [], requirements: [] };
  const match = content.match(/^---\n([\s\S]*?)\n---/);
  if (!match) return result;

  const yaml = match[1];
  const waveMatch = yaml.match(/^wave:\s*(\d+)/m);
  if (waveMatch) result.wave = parseInt(waveMatch[1], 10);

  const depsMatch = yaml.match(/^depends_on:\s*\[([^\]]*)\]/m);
  if (depsMatch) {
    const raw = depsMatch[1].trim();
    if (raw) {
      result.dependsOn = raw.split(',').map(s => s.trim().replace(/^["']|["']$/g, '')).filter(Boolean);
    }
  }

  // Parse requirements array (multi-line YAML list format):
  // requirements:
  //   - DASH-01
  //   - DASH-02
  const reqSection = yaml.match(/^requirements:\s*\n((?:\s+-\s+\S+\n?)+)/m);
  if (reqSection) {
    result.requirements = reqSection[1]
      .split('\n')
      .map(line => line.replace(/^\s+-\s+/, '').trim())
      .filter(Boolean);
  }

  return result;
}
```

This must be propagated through `buildPlanDetails()` â†’ planDetails array â†’ API response â†’ frontend.

## Common Pitfalls

### Pitfall 1: Timeline re-render loses source path
**What goes wrong:** `renderTimeline()` stores `sourcePath` on `container.dataset.sourcePath`. If timeline is re-rendered after branch switch, the stored path may be stale.
**Why it happens:** `selectPhase()` reads `container.dataset.sourcePath` for file loading.
**How to avoid:** Always pass `sourcePath` explicitly through the grouping logic to `renderTimeline()` and `selectPhase()`. Check that the groupPhases refactor passes sourcePath through to each node's click handler.

### Pitfall 2: Sub-phase connector layout breaks with flex-wrap
**What goes wrong:** If the phase timeline wraps to multiple rows (DASH-07 strategy), connectors between parent phase and sub-phase (visual branch) need to handle both same-row and wrapped-row cases.
**Why it happens:** CSS connectors are positioned relative to sibling elements in a flex row.
**How to avoid:** Use the `phase-subphase-group` wrapper div pattern (Pattern 2). The group wraps as a unit, preventing a parent phase from wrapping to a different row than its sub-phases.

### Pitfall 3: Requirements field not in planDetails breaks existing plan rendering
**What goes wrong:** Adding `requirements: []` to `parsePlanFrontmatter` return value is safe (additive), but if the regex fails on older plan format, `requirements` may be undefined in some objects.
**Why it happens:** Older PLAN.md files may not have a `requirements:` block.
**How to avoid:** Default to `[]` if regex match is null. The frontend should always guard: `(detail.requirements || [])`.

### Pitfall 4: innerHTML use for meta card creates XSS risk with plan frontmatter values
**What goes wrong:** Plan frontmatter values (wave numbers, requirement IDs like "DASH-01") are trusted but technically untrusted input.
**Why it happens:** The existing `plan-meta-bar` uses string concatenation with `escapeHtml()`. If the new card uses `innerHTML` with unescaped values, XSS is possible.
**How to avoid:** Build the meta card with `document.createElement()` calls, not innerHTML. This is consistent with the rest of the project's DOM construction pattern.

### Pitfall 5: `Object.keys(dependencies)` may miss phases not in ROADMAP.md
**What goes wrong:** If a source has no ROADMAP.md, `dependencies` is `{}`. Pending phase indicators won't appear for those sources.
**Why it happens:** `parseRoadmapDeps()` returns `{}` if ROADMAP.md is absent.
**How to avoid:** Graceful fallback â€” if `dependencies` is empty, don't show pending indicators (don't crash). The mini-timeline still shows existing phases correctly.

## Open Questions

1. **Threshold for timeline collapse (DASH-07)**
   - What we know: CSS flex-wrap prevents overflow; completed phases can collapse to dot-only
   - What's unclear: Should collapse be always-on (all complete phases dot-only) or triggered by overflow detection? Always-on is simpler to implement and arguably better UX.
   - Recommendation: Always collapse complete phases to dot-only in the full timeline. Selected phase (even if complete) shows full label.

2. **Sub-phase grouping when parent phase doesn't exist yet (DASH-08)**
   - What we know: Phase 4.5 is `complete`, phases 4.5.1 directories exist. Edge case: what if the filesystem has 4.5.1 but not 4.5?
   - What's unclear: Whether to handle this defensively or assume phase files follow GSD convention.
   - Recommendation: Fallback â€” if parent group not found, render sub-phase as peer (already noted in Pattern 2).

3. **Requirements field parsing: inline array vs multi-line list (DASH-12)**
   - What we know: Existing `depends_on: ["04.5-01"]` uses inline JSON array. The `requirements:` field in plan files uses multi-line YAML list (each item on its own line with `- ` prefix).
   - What's unclear: Whether any plan files use inline array for requirements.
   - Recommendation: Support both formats in the regex. Check with: `grep -r "requirements:" .planning/phases/` to confirm format in existing files.
